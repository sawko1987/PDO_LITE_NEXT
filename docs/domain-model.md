# Доменная модель v1

## 1. Назначение
Документ фиксирует согласованное ядро первой версии `PDO_LITE_NEXT` и служит источником истины для `packages/domain`, backend-контрактов и клиентских сценариев.

## 2. Границы v1
- Официальный пользовательский набор типов структуры: машина, место, узел, деталь.
- Операции существуют как отдельный домен, связанный со структурными вхождениями.
- Материалы и спецпроцессы могут присутствовать в импорте как технические данные, но не становятся самостоятельными пользовательскими узлами структуры v1.
- Минимальная единица планирования: конкретное `structure occurrence` версии машины.
- Задания мастерам выпускаются по операциям, принадлежащим этим occurrence.
- Источник истины: локальная доменная модель и локальный backend, а не состояние мобильного клиента.

## 3. Словарь сущностей

### Machine
- Идентификатор: `machineId`.
- Обязательные поля: код, имя.
- Связи: имеет много `MachineVersion`.
- Правило: активная версия является ссылкой, а не заменой истории.

### MachineVersion
- Идентификатор: `versionId`.
- Обязательные поля: `machineId`, `label`, `createdAt`, `status`.
- Статусы: `draft`, `published`, `archived`.
- Правило: опубликованная версия неизменяема; любые изменения структуры и операций создают новую версию.

### CatalogItem
- Идентификатор: `catalogItemId`.
- Обязательные поля: код, имя, тип.
- Назначение: справочная карточка номенклатурного объекта, которая может встречаться в нескольких местах структуры.

### StructureOccurrence
- Идентификатор: `structureOccurrenceId`.
- Обязательные поля: `versionId`, `catalogItemId`, `pathKey`, `displayName`, `quantityPerMachine`.
- Связи: принадлежит одной версии машины, может иметь родителя.
- Правило: одинаковый `catalogItemId` может встречаться в нескольких местах версии и не схлопывается.

### OperationOccurrence
- Идентификатор: `operationOccurrenceId`.
- Обязательные поля: `versionId`, `structureOccurrenceId`, `name`, `quantityPerMachine`.
- Связи: принадлежит конкретному структурному вхождению.
- Правило: задания выпускаются по `OperationOccurrence`, а не по свободному тексту.

### Plan
- Идентификатор: `planId`.
- Обязательные поля: `machineId`, `versionId`, `title`, `createdAt`, `status`.
- Статусы: `draft`, `released`, `completed`, `cancelled`.
- Правило: план всегда ссылается на конкретную версию машины.

### PlanItem
- Идентификатор: `planItemId`.
- Обязательные поля: источник (`PlanItemSource`), `requestedQuantity`.
- Правило: источник плановой позиции обязан ссылаться на конкретный `StructureOccurrence`.

### PlanRevision
- Идентификатор: `planRevisionId`.
- Обязательные поля: `planId`, `revisionNumber`, `changedBy`, `changedAt`, список diff-изменений.
- Правило: изменения выпущенного плана фиксируются с детализацией "до/после".

### ProductionTask
- Идентификатор: `taskId`.
- Обязательные поля: `planItemId`, `operationOccurrenceId`, `requiredQuantity`, `status`.
- Статусы: `pending`, `inProgress`, `completed`, `cancelled`.
- Правило: задание считается закрытым только в статусах `completed` или `cancelled`.

### ExecutionReport
- Идентификатор: `executionReportId`.
- Обязательные поля: `taskId`, `reportedBy`, `reportedAt`, `reportedQuantity`.
- Назначение: фиксирует факт выполнения по конкретному заданию.
- Правило v1: execution report идемпотентен по `requestId`, перевыполнение сверх `remainingQuantity` запрещено.

### Problem
- Идентификатор: `problemId`.
- Обязательные поля: `machineId`, `type`, `status`, `createdAt`.
- Дополнительные поля: `taskId`, `title`.
- Статусы v1: `open`, `inProgress`, `closed`.
- Правило v1: проблема создаётся из задания и закрывается без отдельного supervisor-step.

### ProblemType
- Допустимые значения:
  - `equipment`
  - `materials`
  - `documentation`
  - `planningError`
  - `technologyError`
  - `blockedByOtherWorkshop`
  - `other`

### ProblemMessage
- Идентификатор: `problemMessageId`.
- Обязательные поля: `problemId`, `authorId`, `message`, `createdAt`.
- Назначение: хранит сообщения problem-thread.
- Правило v1: закрытая проблема не принимает новые сообщения.

### WipEntry
- Идентификатор: `wipEntryId`.
- Обязательные поля: `machineId`, `versionId`, `structureOccurrenceId`, `operationOccurrenceId`, `balanceQuantity`, `status`.
- Статусы: `open`, `transferred`, `consumed`, `writtenOff`.
- Правило: открытый НЗП с положительным остатком блокирует завершение изделия.

### UserRole
- Официальный набор ролей v1: `planner`, `supervisor`, `master`.

### AuditEntry
- Идентификатор: `auditEntryId`.
- Обязательные поля: тип сущности, идентификатор сущности, действие, автор, время.
- Назначение: хранит критичные изменения и diff там, где это требуется.

## 4. Правила жизненного цикла

### Версии машины
- `draft -> published -> archived`
- Переход назад из `published` не предусмотрен.
- Правка опубликованной версии запрещена.

### Планы
- `draft -> released -> completed`
- `draft -> cancelled`
- `released -> cancelled` допускается только отдельным backend-решением и без переписывания принятого факта.
- Выпущенный план можно редактировать только в частях без зафиксированного факта выполнения.

### Задания
- `pending -> inProgress -> completed`
- `pending -> cancelled`
- `inProgress -> cancelled` требует отдельного аудита.

### Проблемы
- `open -> inProgress -> closed`
- `open -> closed` допустим как упрощённый v1-сценарий.
- Возврат из `closed` не поддерживается в v1.
- Закрытая проблема не принимает новые сообщения.

### НЗП
- `open -> transferred`
- `open -> consumed`
- `open -> writtenOff`
- Закрывающие статусы не должны сохранять положительный остаток как активный НЗП.

## 5. Ключевые бизнес-правила v1
- Структура и планирование строятся по occurrence, а не по абстрактному коду детали.
- Старая версия машины не меняется при появлении новой.
- Уже выпущенный план нельзя переписывать в части, где есть факт выполнения.
- Изменения выпущенного плана обязаны сохранять автора, время и diff "до/после".
- Execution report меняет статус задания `pending -> inProgress -> completed`.
- Изделие можно завершить только если:
  - все задания закрыты;
  - нет открытых проблем;
  - нет открытого НЗП с положительным остатком.
- В текущем v1 problem flow является task-scoped:
  - проблема создаётся из карточки задания;
  - `machineId` выводится из задания;
  - мастер может создать проблему, писать в чат и закрыть её сам;
  - supervisor confirmation для закрытия проблемы в v1 не требуется.

## 6. Что остаётся вне этого этапа
- физическая схема БД и миграции;
- полноценная модель пользователей и авторизации;
- полный модуль НЗП с переносом между планами;
- reopen проблемы;
- вложения и фото;
- sync-engine с фоновыми retry и разбором конфликтов.
