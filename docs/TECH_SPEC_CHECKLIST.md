# ТЗ: новое производственное решение на Flutter

## Summary
- [ ] Создать новую систему с нуля, не переписывая текущий Flask/Tkinter-код и не перенося старые данные из существующей БД.
- [ ] Построить решение как один backend + два Flutter-клиента.
- [ ] Сделать Flutter Windows для планирования, управления производством и локального хранения.
- [ ] Сделать Flutter mobile для мастеров.
- [ ] Сделать локальную БД на ПК главным источником истины.
- [ ] Использовать облачный контур только как временный связующий слой для телефонов, а не как постоянное хранилище истории.
- [x] Заложить чистую архитектуру по доменам: Номенклатура, Версии машин, Планы, Задания, Факт, НЗП, Проблемы, Отчёты, Пользователи.
- [x] Зафиксировать режим `desktop-first`: сначала довести `backend + admin_windows` до полного `desktop-v1`, и только потом расширять `master_mobile` и полный sync-контур.
- [x] Зафиксировать правило: после каждого завершённого этапа обновлять roadmap и checklist в том же change set.
- [ ] Исключить перенос архитектурных дефектов текущей системы: монолитный backend, двусмысленные связи, неявные миграции, смешение оперативных и архивных данных, ручные фиксы скриптами.

## Импорт, синхронизация и качество
- [x] Спроектировать импорт Excel/MXL как отдельный модуль с этапами: загрузка, разбор, валидация, предпросмотр, подтверждение, создание версии машины.
- [x] Исключить тихие автоправки импортируемой структуры.
- [x] Показывать пользователю конфликтные места импорта до сохранения.
- [ ] Ввести правила валидации: обязательные поля, уникальность кодов в нужной области, корректность иерархии, отсутствие циклов, корректность операций и цехов.
- [ ] Спроектировать синхронизацию как однонаправленную по модели истины: ПК-сервер публикует активные задания и справочники в облачный контур, мобильные клиенты читают и отправляют факт, сервер подтверждает, журналирует и архивирует.
- [ ] Очистку облака делать не физическим удалением всего подряд, а управляемым жизненным циклом активных данных.
- [ ] Исключить прямую зависимость бизнес-логики от облачного контура.
- [ ] Добавить мониторинг синхронизации: очередь, ошибки, повторные попытки, зависшие события, конфликтные записи.
- [ ] Заложить резервное копирование локальной БД и восстановление из бэкапа.
- [ ] Разделить активные, архивные и служебные данные.
