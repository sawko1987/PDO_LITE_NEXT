# ТЗ: новое производственное решение на Flutter

## Summary
- [ ] Создать новую систему с нуля, не переписывая текущий Flask/Tkinter-код и не перенося старые данные из существующей БД.
- [ ] Построить решение как один backend + два Flutter-клиента.
- [ ] Сделать Flutter Windows для планирования, управления производством и локального хранения.
- [ ] Сделать Flutter mobile для мастеров.
- [ ] Сделать локальную БД на ПК главным источником истины.
- [ ] Использовать облачный контур только как временный связующий слой для телефонов, а не как постоянное хранилище истории.
- [x] Заложить чистую архитектуру по доменам: Номенклатура, Версии машин, Планы, Задания, Факт, НЗП, Проблемы, Отчёты, Пользователи.
- [ ] Исключить перенос архитектурных дефектов текущей системы: монолитный backend, двусмысленные связи, неявные миграции, смешение оперативных и архивных данных, ручные фиксы скриптами.

## Архитектура и данные
- [ ] Спроектировать новый backend API для локального сервера/ПК.
- [ ] Выбрать нормализованную основную БД для ПК-сервера.
- [ ] Зафиксировать правило: локальная БД хранит полную историю бессрочно, облачный контур хранит только активные данные для мастеров.
- [ ] Реализовать сущности: Machine, MachineVersion, StructureItem, Operation, Plan, PlanItem, Task, Report, WipEntry, Problem, ProblemMessage, User, Role, Workshop.
- [ ] Ввести неизменяемые идентификаторы и аудит изменений для всех критичных сущностей.
- [x] Хранить версии машин отдельно от планов: старые планы не меняются при появлении новой версии.
- [ ] Разрешить создание новой версии машины на основе предыдущей.
- [ ] Разрешить добавление в план всей машины целиком, отдельного узла, отдельного места, отдельной детали.
- [ ] Реализовать контролируемое обновление структуры машины без переписывания истории старых планов.
- [ ] Вести полный аудит: кто создал, кто изменил, когда, что было до/после.
- [x] Разделить статусы на уровне машины, плана, задания, проблемы и НЗП.

## Функциональность первого релиза
- [ ] Реализовать роли первого релиза: планировщик, начальник, мастер.
- [ ] Сделать вход по логину/паролю.
- [ ] Для ПК-клиента реализовать модули: справочник машин, версии машин, импорт Excel/MXL, редактор структуры машины, формирование плана, выпуск заданий мастерам по операциям, контроль выполнения, подтверждение завершения изделия начальником, учёт НЗП, проблемы и журнал отклонений, отчёты план-факт, сменные, проблемы.
- [ ] Для мобильного клиента мастера реализовать: список заданий, фильтры по статусу/машине/номенклатуре, просмотр операции и контекста номенклатуры, ввод факта выполнения, фиксацию невыполнения/частичного выполнения, создание проблемы, чат по проблеме, офлайн-очередь факта и проблем, последующую синхронизацию.
- [ ] Реализовать полный учёт НЗП: остаток по операции, перевыполнение, перенос НЗП в последующие планы, журнал движения НЗП.
- [ ] Считать изделие завершённым только после подтверждения начальником.
- [ ] После завершения изделия убирать его из облачного рабочего контура, но оставлять всю историю локально.
- [ ] Не делать в первом релизе вложения/фото.

## Импорт, синхронизация и качество
- [ ] Спроектировать импорт Excel/MXL как отдельный модуль с этапами: загрузка, разбор, валидация, предпросмотр, подтверждение, создание версии машины.
- [ ] Исключить тихие автоправки импортируемой структуры.
- [ ] Показывать пользователю конфликтные места импорта до сохранения.
- [ ] Ввести правила валидации: обязательные поля, уникальность кодов в нужной области, корректность иерархии, отсутствие циклов, корректность операций и цехов.
- [ ] Спроектировать синхронизацию как однонаправленную по модели истины: ПК-сервер публикует активные задания и справочники в облачный контур, мобильные клиенты читают и отправляют факт, сервер подтверждает, журналирует и архивирует.
- [ ] Очистку облака делать не физическим удалением всего подряд, а управляемым жизненным циклом активных данных.
- [ ] Исключить прямую зависимость бизнес-логики от облачного контура.
- [ ] Добавить мониторинг синхронизации: очередь, ошибки, повторные попытки, зависшие события, конфликтные записи.
- [ ] Заложить резервное копирование локальной БД и восстановление из бэкапа.
- [ ] Разделить активные, архивные и служебные данные.

- [ ] Убрать ручные диагностические скрипты как обязательную часть эксплуатации; заменить штатной диагностикой в системе.
- [ ] Убрать неявные пересоздания сущностей на лету при создании заданий.
- [x] Ввести явные состояния и бизнес-правила для версии машины, плана, задания и закрытия изделия.
- [ ] Ввести обязательный аудит и защиту от потери и перемешивания данных при конкурентной работе.

- [ ] Тест: план можно сформировать по машине целиком и по отдельным компонентам.
- [ ] Тест: задания мастерам создаются именно по операциям.
- [ ] Тест: частичное выполнение корректно формирует остаток.
- [ ] Тест: перевыполнение уходит в НЗП и не теряется.
- [ ] Тест: завершённое изделие уходит из облачного контура только после подтверждения начальником.

## Assumptions
- [ ] Основная панель — Flutter Windows.
- [ ] Мастерский клиент — Flutter mobile.
- [ ] Старые данные из текущей БД не переносим.
- [ ] Источник истины — локальная БД на ПК/сервере.
- [ ] Облако используется как временный транспортный слой для телефонов.
- [ ] Номенклатура машин хранится как справочник с версиями.
- [ ] Первая версия продукта не включает вложения/фото.
- [ ] Первый релиз покрывает админскую часть и работу мастеров, но без переноса всей старой отчетности и без повторения старых ошибок.
