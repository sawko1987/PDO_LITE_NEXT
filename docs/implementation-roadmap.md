# Дорожная карта реализации

## 1. Назначение
Этот документ фиксирует последовательность разработки нового производственного решения и нужен для того, чтобы проект не расползся по старому сценарию с постоянными переделками и смешением логики.

Главная цель:
- сначала построить правильное ядро;
- потом подключать интерфейсы;
- потом вводить синхронизацию и эксплуатационные функции;
- не начинать UI раньше, чем зафиксированы домен и правила данных.

## 2. Общий порядок этапов
Рекомендуемый порядок реализации:
1. Анализ legacy-проекта
2. Проектирование доменной модели
3. Проектирование БД и API
4. Реализация модуля импорта Excel/MXL
5. Реализация локального backend
6. Реализация Windows-приложения
7. Реализация мобильного приложения мастера
8. Реализация синхронизационного контура
9. Отчёты, аудит и резервное копирование
10. Интеграционное тестирование
11. Пилотный запуск
12. Доработки по результатам пилота

- [ ] Найти все правила по проблемам и статусам.

- [ ] Подтверждён список бизнес-правил.

- [ ] Не начинать UI до фиксации доменной модели.
- [ ] Не переносить старые таблицы и хаотичные миграции.
- [ ] Не делать облако главным источником истины.
- [ ] Не копировать старый парсинг без формализации.
- [ ] Не выпускать первый релиз без проверки НЗП и завершения изделия.
- [ ] Не удалять активные данные из облака без подтверждения, что они приняты локально.
- [ ] Не запускать пилот без резервного копирования и диагностики.
