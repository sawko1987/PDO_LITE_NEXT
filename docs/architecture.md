# Архитектура нового производственного решения

## 1. Назначение
Новая система создаётся как отдельный продукт для управления производством и работы мастеров без наследования архитектурных ошибок текущего проекта.

Система должна:
- хранить полную историю локально на ПК;
- поддерживать работу мастеров с телефона;
- обеспечивать планирование по машинам, узлам, местам, деталям и операциям;
- поддерживать версии машин;
- вести НЗП, проблемы, отчётность и аудит;
- использовать облачный контур только как транспортный слой для мобильных клиентов.

## 2. Общая схема решения
Решение состоит из трёх основных частей:

1. `Admin Windows App`
- основной клиент для планировщика и начальника;
- работает на Windows;
- подключается к локальному backend;
- используется для импорта, ведения справочников, планирования, контроля и отчётов.

2. `Local Backend`
- локальный сервер на том же ПК или в локальной сети;
- предоставляет API для Windows-клиента и мобильного клиента;
- работает с основной локальной БД;
- управляет импортом, версиями машин, планами, заданиями, НЗП, проблемами, аудитом и синхронизацией.

3. `Master Mobile App`
- мобильное приложение для мастеров;
- получает задания и справочники через синхронизационный контур;
- поддерживает офлайн-работу;
- отправляет факт выполнения и проблемы.

## 3. Принцип источника истины
### Главный источник истины
Главным источником истины является:
- локальная БД на ПК/сервере.

Именно локальная БД хранит:
- машины;
- версии машин;
- структуру изделий;
- операции;
- планы;
- задания;
- фактическое выполнение;
- НЗП;
- проблемы;
- историю;
- аудит;
- архив.

### Облачный контур
Облачный контур не является главным хранилищем.

Он используется только для:
- доставки активных заданий на телефоны;
- передачи факта выполнения обратно;
- временного обмена рабочими данными;
- обслуживания мобильных сценариев вне локальной сети.

### Ограничение
После завершения изделия и подтверждения начальником:
- активные данные по нему должны быть удалены из облачного контура;
- локальная история должна сохраняться полностью.

## 4. Логическое разделение по приложениям
### 4.1 Admin Windows App
Роли:
- планировщик;
- начальник.

Основные функции:
- импорт Excel/MXL;
- создание и ведение машин;
- создание версий машин;
- редактирование структуры;
- формирование планов;
- выпуск заданий;
- контроль выполнения;
- подтверждение завершения изделий;
- работа с НЗП;
- управление проблемами;
- просмотр отчётов;
- работа с архивом;
- администрирование пользователей.

### 4.2 Master Mobile App
Роль:
- мастер.

Основные функции:
- получение заданий;
- просмотр контекста машины и номенклатуры;
- ввод выполненного количества;
- фиксация частичного выполнения;
- фиксация проблемы;
- чат по проблеме;
- офлайн-очередь действий;
- синхронизация после появления сети.

## 5. Основные домены системы
### 5.1 Машины
Сущность машины описывает изделие верхнего уровня.

Хранит:
- имя;
- код;
- описание;
- служебные признаки;
- текущую активную версию;
- архив версий.

### 5.2 Версии машин
Каждая версия машины фиксирует структуру на определённый момент.

Правила:
- версия неизменяема после публикации;
- новая загрузка изменённой структуры создаёт новую версию;
- старые планы остаются привязаны к старой версии;
- новые планы могут использовать новую версию.

### 5.3 Структура изделия
Структура включает:
- места;
- узлы;
- детали;
- другие допустимые уровни.

Каждый элемент структуры должен иметь:
- тип;
- родителя;
- уровень;
- код;
- количество;
- цех;
- служебный raw-payload для диагностики импорта.

### 5.4 Операции
Операции существуют как отдельные сущности, связанные с элементами структуры.

Хранят:
- название;
- цех;
- порядок выполнения;
- нормы;
- количество;
- связь с номенклатурой.

### 5.5 Планы
План описывает производственный запуск.

Хранит:
- ссылку на машину;
- ссылку на конкретную версию;
- период;
- статус;
- состав плановых позиций;
- ответственных;
- историю изменений.

### 5.6 Задания
Задания выдаются мастерам по операциям.

Хранят:
- план;
- машину;
- версию;
- номенклатуру;
- операцию;
- цех;
- мастера;
- требуемое количество;
- выполненное количество;
- статус;
- срок;
- приоритет.

### 5.7 Факт выполнения
Факт выполнения создаётся мастером и подтверждается сервером.

Хранит:
- ссылку на задание;
- количество;
- время;
- автора;
- причину частичного выполнения или невыполнения;
- отметку синхронизации.

### 5.8 НЗП
НЗП хранится как отдельный домен.

Хранит:
- машину;
- версию;
- номенклатуру;
- операцию;
- остаток;
- перевыполнение;
- источник возникновения;
- движение;
- статус использования.

### 5.9 Проблемы
Проблемы оформляются как отдельные инциденты.

Хранят:
- тип проблемы;
- описание;
- связанное задание или изделие;
- статус;
- автора;
- ответственного;
- сообщения чата;
- историю жизненного цикла.

### 5.10 Аудит
Аудит обязателен для критичных сущностей.

Хранит:
- кто изменил;
- когда изменил;
- сущность;
- поле;
- старое значение;
- новое значение;
- тип действия.

## 6. Модель хранения данных
### Основная БД
Основная БД должна хранить:
- оперативные данные;
- архив;
- версии;
- аудит;
- системные журналы;
- очередь синхронизации;
- настройки.

### Временные данные облачного контура
Во внешнем контуре допустимо хранить только:
- активные задания;
- ограниченный набор справочных данных для мастеров;
- неподтверждённые отчёты до приёма сервером;
- временные данные синхронизации.

### Недопустимо
Нельзя хранить в облаке как единственный экземпляр:
- историю версий;
- архив изделий;
- единственный экземпляр НЗП;
- единственный экземпляр проблем;
- аудит.

## 7. Синхронизация
### Принцип
Синхронизация строится от локального сервера наружу.

Основной поток:
1. Локальный backend формирует активный набор данных для мастеров.
2. Эти данные отправляются в облачный транспортный контур.
3. Мобильный клиент получает задания и справочники.
4. Мастер отправляет факт выполнения или проблему.
5. Локальный backend принимает событие, валидирует его и сохраняет в локальной БД.
6. После завершения изделия активные облачные данные очищаются.

### Требования
- синхронизация не должна менять локальную историю;
- все операции должны логироваться;
- должны быть повторные попытки;
- должны быть статусы очереди;
- должны быть инструменты диагностики;
- должны быть безопасные правила очистки облака.

### Офлайн
Мобильный клиент должен:
- кэшировать задания;
- кэшировать нужные справочники;
- хранить локальную очередь отчётов и проблем;
- синхронизировать их при восстановлении связи.

## 8. Импорт Excel/MXL
Импорт должен быть вынесен в отдельный модуль.

### Этапы импорта
1. Загрузка файла.
2. Определение формата.
3. Парсинг.
4. Нормализация.
5. Построение иерархии.
6. Валидация.
7. Предпросмотр.
8. Подтверждение пользователем.
9. Создание новой машины или новой версии.

### Требования
- никакой записи в боевую БД до подтверждения;
- никакой скрытой правки данных;
- предупреждения и ошибки должны быть видны;
- результат импорта должен быть тестируемым отдельно от UI и БД.

## 9. Структура репозитория
Рекомендуемая структура:

```text
D:\PDO_LITE_NEXT
├─ docs
│  ├─ TECH_SPEC_CHECKLIST.md
│  ├─ architecture.md
│  └─ legacy-analysis
│     ├─ parsing_excel_mxl.md
│     └─ field_mapping_template.md
├─ apps
│  ├─ admin_windows
│  └─ master_mobile
├─ backend
├─ packages
│  ├─ domain
│  ├─ data_models
│  ├─ import_engine
│  └─ shared_ui
└─ tools
```

### Назначение папок
- `apps/admin_windows` — Flutter Windows клиент.
- `apps/master_mobile` — Flutter mobile клиент.
- `backend` — локальный сервер и бизнес-логика.
- `packages/domain` — чистые доменные сущности и правила.
- `packages/data_models` — DTO, схемы, сериализация.
- `packages/import_engine` — импорт Excel/MXL.
- `packages/shared_ui` — общие UI-компоненты.
- `tools` — служебные скрипты нового проекта.

## 10. Технические принципы
- разделять UI, домен, инфраструктуру и синхронизацию;
- не смешивать импорт и запись в БД;
- не использовать runtime-хаос со схемой БД;
- применять управляемые миграции;
- делать все критичные операции идемпотентными;
- журналировать синхронизацию;
- проектировать явные статусы;
- не использовать облако как архив;
- не переносить legacy-костыли в новый код.

## 11. Первый релиз
Первый релиз должен включать:
- Windows-панель для планировщика и начальника;
- мобильное приложение мастера;
- локальный backend;
- локальную БД;
- импорт Excel/MXL;
- версии машин;
- планирование по операциям;
- задания мастерам;
- факт выполнения;
- НЗП;
- проблемы;
- отчёты план-факт, сменные, проблемы;
- аудит;
- синхронизацию мобильного контура.
