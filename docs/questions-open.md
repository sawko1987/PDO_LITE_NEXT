### NOM-003. Нужно ли ручное редактирование структуры после импорта
Статус:
- [ ] открыт
- [x] закрыт

Описание:
- импорт будет основным способом загрузки;
- не ясно, насколько глубоко пользователь сможет править структуру вручную.

Что нужно решить:
- разрешено ли ручное редактирование;
- можно ли добавлять узлы и детали вручную;
- можно ли менять операции вручную;
- можно ли менять структуру опубликованной версии.

Решение:
- в v1 ручное редактирование структуры разрешено только в `draft/new version`;
- опубликованные (`published`) версии неизменяемы и доступны только для чтения;
- для правки опубликованной версии пользователь сначала создаёт новую черновую версию на её основе;
- в черновике разрешены add/edit/delete для structure occurrences и operation occurrences;
- после завершения правок черновик публикуется как новая active version.

## 5. Вопросы по планированию
### PLAN-001. Какой объект является минимальной единицей планирования
Статус:
- [ ] открыт
- [x] закрыт

Описание:
- сейчас согласовано, что мастерам задания идут по операциям;
- нужно окончательно зафиксировать минимальный уровень планирования в модели.

Что нужно решить:
- план всегда строится от версии машины;
- можно ли планировать отдельную деталь вне полной машины;
- можно ли планировать только узел;
- можно ли включать НЗП как отдельную плановую сущность.

Решение:
- минимальная единица планирования в v1 — конкретное `structure occurrence` версии машины;
- план обязан ссылаться на конкретную версию машины;
- задания мастерам выпускаются по операциям, принадлежащим этим occurrence.

### PLAN-002. Можно ли редактировать уже выпущенный план
Статус:
- [ ] открыт
- [x] закрыт

Описание:
- после выпуска заданий изменение плана может быть рискованным;
- нужно описать строгие правила.

Что нужно решить:
- можно ли менять состав плана;
- можно ли менять количество;
- можно ли менять мастеров;
- как фиксируется история изменений;
- когда нужен перевыпуск плана.

Решение:
- выпущенный план можно менять только в тех частях, по которым ещё нет факта выполнения;
- части с уже принятым фактом выполнения блокируются от прямого редактирования;
- каждое изменение выпущенного плана должно сохранять автора, время и diff "до/после".

## 6. Вопросы по НЗП
### WIP-001. Как именно переносить НЗП в следующий план
Статус:
- [ ] открыт
- [x] закрыт

Описание:
- полный учёт НЗП обязателен;
- нужно зафиксировать механизм переноса.

Что нужно решить:
- перенос вручную или автоматически;
- переносится ли остаток по операции;
- переносится ли перевыполнение;
- как исключать повторный учёт;
- как отображать происхождение НЗП.

Решение:
- в текущем этапе сервер автоматически ведёт остатки НЗП по результату execution report;
- перенос НЗП в следующий план не делается автоматически;
- подключение НЗП к следующему плану остаётся явным действием планировщика;
- partial, not completed и overrun фиксируются через явный outcome execution report, а не выводятся неявно только из количества.

### PROB-002. Кто может закрывать проблему
Статус:
- [ ] открыт
- [x] закрыт

- есть чат и статусы;
- нужно определить правило подтверждения закрытия.
