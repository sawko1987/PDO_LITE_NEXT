# Анализ и перенос парсинга Excel/MXL

## Цель
Этот документ фиксирует, как подойти к переносу парсинга из старого проекта в новую систему без копирования старых архитектурных ошибок.

Главный принцип:
- старый код парсинга используется как источник знаний;
- в новый проект переносится не весь старый код, а только подтверждённая логика;
- сначала описывается фактическое поведение парсера, затем проектируется новый модуль импорта.

## Что уже известно по legacy-проекту
В старом проекте парсинг завязан минимум на следующие части:
- `excel_parser.py`
- `mxl_parser.py`
- `tk_app/product_importer.py`
- `database.py`
- маршруты загрузки и импорта в `app.py`

Из текущей структуры видно:
- парсер читает файл изделия;
- выделяет информацию о машине;
- выделяет компоненты структуры;
- пытается восстановить иерархию;
- сохраняет результат в локальную БД;
- часть структуры определяется не только содержимым строки, но и положением строки в исходном файле;
- есть отдельная логика для `object_type`, `parent_id`, `level`, `position_number`, `parent_name`, `parent_code`, `raw_data`;
- есть различие между номенклатурой, материалами и операциями;
- есть риск неявного достраивания связей при импорте.

## Главные проблемы старого парсинга
- логика парсинга и логика сохранения в БД сцеплены слишком тесно;
- часть смысла извлекается эвристически, а не по строгой модели;
- иерархия может восстанавливаться по близости строк, а не по надёжным ключам;
- структура после импорта зависит от текущей схемы БД;
- типы сущностей частично определяются косвенно;
- нет жёсткого этапа валидации перед записью;
- нет формального контракта результата парсинга;
- нет отдельного режима предпросмотра без сохранения.

## Что нужно выяснить при анализе legacy-парсинга
- какие форматы файлов реально используются;
- какие шаблоны файлов приходят чаще всего;
- какие поля машины извлекаются обязательно;
- какие поля компонента извлекаются обязательно;
- как определяется тип строки;
- как определяется уровень вложенности;
- как определяется родительский элемент;
- как определяется позиционный номер;
- как определяется цех;
- как определяется количество;
- как определяется код;
- как определяется связь операции с номенклатурой;
- какие данные игнорируются;
- какие ошибки импорта встречаются чаще всего.

## Что нужно сохранить из старого поведения
- поддержку Excel и MXL;
- извлечение карточки машины;
- извлечение структуры изделия;
- различение номенклатуры, материалов и операций;
- восстановление иерархии;
- поддержку цехов;
- поддержку количества и кодов;
- возможность повторной загрузки изменённой машины.

## Что нельзя переносить в новом виде
- прямую запись в боевую БД во время сырого разбора;
- неявные автоправки структуры без показа пользователю;
- смешение этапов разобрать, проверить, сохранить;
- зависимость модели импорта от старой схемы SQLite;
- runtime-миграции как часть импорта;
- создание побочных сущностей на лету без подтверждения;
- скрытую денормализацию как основу модели.

## Как должен выглядеть новый модуль импорта
### Общая схема
- `Loader` — читает файл и определяет формат.
- `Parser` — извлекает сырые строки и поля.
- `Normalizer` — приводит данные к единой внутренней модели.
- `HierarchyBuilder` — строит дерево структуры.
- `Validator` — проверяет корректность.
- `PreviewBuilder` — готовит результат для экрана предпросмотра.
- `Importer` — только после подтверждения создаёт новую версию машины.

### Обязательные этапы
- загрузка файла;
- определение формата;
- парсинг в промежуточную модель;
- нормализация;
- валидация;
- показ предупреждений и конфликтов;
- подтверждение пользователем;
- создание версии машины;
- журнал результата импорта.

## Контракт нового результата парсинга
Новый парсер должен возвращать не записи БД, а отдельную импортную модель, например:
- `machine`
- `version_meta`
- `items[]`
- `operations[]`
- `materials[]`
- `warnings[]`
- `errors[]`
- `source_file_info`

### Обязательные поля `machine`
- `name`
- `code`
- `source_file_name`
- `imported_at`

### Обязательные поля `item`
- `external_row_id`
- `name`
- `code`
- `item_type`
- `level`
- `parent_external_row_id`
- `position_number`
- `workshop`
- `quantity`
- `raw_payload`

### Обязательные поля `operation`
- `external_row_id`
- `related_item_external_row_id`
- `name`
- `workshop`
- `operation_order`
- `quantity`
- `raw_payload`

## Правила валидации нового импорта
- у машины должно быть имя;
- у каждого элемента должен быть определён тип;
- иерархия не должна содержать циклы;
- у дочернего элемента должен быть корректный родитель или допустимый корень;
- операция должна быть привязана к допустимой номенклатуре;
- количество должно быть числом и входить в допустимый диапазон;
- обязательные колонки файла должны быть обнаружены;
- дубли кодов должны помечаться явно;
- строки с неоднозначной интерпретацией должны попадать в предупреждения, а не сохраняться молча.

## Экран предпросмотра импорта
Перед сохранением пользователь должен видеть:
- карточку машины;
- предполагаемое имя версии;
- количество элементов структуры;
- количество операций;
- количество материалов;
- предупреждения;
- ошибки;
- список конфликтных строк;
- итоговое действие: сохранить как новую машину, сохранить как новую версию существующей машины или отменить импорт.

## Правила версионирования машины
- каждая загрузка изменённой структуры создаёт новую версию;
- старая версия не редактируется задним числом;
- новая версия не меняет старые планы;
- пользователь должен явно выбрать: создать новую машину, создать новую версию существующей машины или отклонить импорт.

## План исследования legacy-кода
### Этап 1. Карта текущего парсинга
- найти все входы в импорт;
- определить, какие модули реально участвуют;
- зафиксировать порядок вызовов;
- выписать все поля, которые извлекаются и записываются.

### Этап 2. Полевая модель
- составить таблицу колонка файла -> внутреннее поле;
- зафиксировать обязательные и необязательные поля;
- описать правила распознавания типов строк.

### Этап 3. Иерархия
- отдельно разобрать, как сейчас строится `parent_id`;
- отдельно разобрать логику уровней `level`;
- отдельно разобрать позиционные номера и близость строк;
- отметить все эвристики, которые могут давать ошибки.

### Этап 4. Номенклатура и операции
- понять, как операция связывается с номенклатурой;
- понять, как выделяются материалы;
- понять, какие поля нужны для планирования и выдачи заданий.

### Этап 5. Тестовые образцы
- собрать набор реальных файлов: простой файл, файл со сложной иерархией, файл с изменённой версией машины, файл с проблемной структурой, файл с дубликатами или неоднозначностью;
- для каждого файла сохранить ожидаемый результат разбора.

- автотесты на парсер
- автотесты на построение иерархии
- автотесты на валидацию

## Что считать успешным завершением переноса парсинга
- новый модуль импорта разбирает реальные файлы без зависимости от legacy-БД;
- пользователь видит результат до сохранения;
- ошибки и неоднозначности не скрываются;
- новая версия машины создаётся предсказуемо;
- старые планы не ломаются;
- результаты парсинга покрыты тестами на реальных примерах.

## Решение по внедрению
- сначала анализ и документирование legacy-парсинга;
- затем проектирование нового импортного контракта;
- затем реализация нового парсера в отдельном модуле;
- затем тестирование на реальных файлах;
- только потом подключение к новой БД и UI.
